## 2. True Service Architecture (SOA Target)

The design philosophy: **CCC is the persistent supervisor**. Everything else is a service it manages.

```
CCC (Rust/Axum, port 8789)
├── ccc-server        — Core API: queue, agents, heartbeats, secrets, ClawBus, auth
├── ccc-brain         — LLM request queue + retry (tokio background task in ccc-server)
├── ccc-vector        — Milvus embed/search (tokio background task in ccc-server)
├── dashboard-server  — Static WASM UI server + CCC API proxy
├── squirrelchat-server — Chat (Rust, separate binary, port 8793)
├── tokenhub          — LLM router (Go binary, port 8090, launched by CCC)
└── openclaw          — Self-managed, talks to CCC via API
```

**What CCC manages (process supervisor role):**
- tokenhub binary lifecycle (start/stop/restart/health-check)
- dashboard-server lifecycle
- squirrelchat-server lifecycle
- SSH reverse tunnels to Sweden containers (currently systemd, should be CCC-supervised)

CCC does NOT manage: openclaw (self-managed), systemd itself, Mattermost, Grafana, MinIO.

## 5. Doltlite / Task Server

jkh mentioned "task server based on doltlite." This is **not yet implemented** in the repo.

Current state: work queue lives in `workqueue/queue.json` (flat JSON file). This is a known limitation — concurrent writes, no versioning, no git history.

**Doltlite target architecture:**
- Replace `queue.json` file store with a Dolt database (git-versioned SQL)
- CCC queue API (`/api/queue/*`) talks to Dolt instead of JSON file
- Every queue mutation is a Dolt commit → full audit trail
- Branching enables "draft" work items before publishing

**Current blockers:**
- Dolt binary not installed on do-host1
- No Dolt schema defined for queue items
- ccc-server Rust queue routes use serde_json file I/O — need to add Dolt client

This is a separate work item from the Node→Rust migration.

## 6. Tmux/Claude Code Integration

jkh mentioned "tmux connection to claude code." This refers to OpenClaw's ACP harness — Claude Code runs in a tmux session on the agent host and OpenClaw connects to it via the ACP (Agent Capability Protocol).

Current state in CCC:
- .ccc/api/index.mjs` has `/api/agentos/shell` endpoints — these interact with a terminal
- `services.ccc/skills/tmux/` handles raw tmux interaction
- OpenClaw has a `sessions_spawn(runtime="acp")` path for Claude Code

**Gap:** CCC doesn't have a tmux session manager or Claude Code lifecycle manager. This should be:
1. CCC registers which agent hosts have active Claude Code sessions
2. Dashboard shows Claude Code session status per agent
3. Work items can be dispatched to a Claude Code session via ACP

This is the `coding-agent` skill territory — CCC needs an ACP session registry endpoint (`/api/acp/sessions`).

## 7. openclaw as a Service

OpenClaw is currently self-managed (not supervised by CCC). It should register itself as a service with CCC on startup and send heartbeats. The gateway restart path (`gateway.restart`) should also be reachable via CCC.

CCC already has `GET /api/agents` — openclaw should auto-register as an agent on do-host1 with capabilities: `["chat", "slack", "telegram", "acp", "browser"]`.
